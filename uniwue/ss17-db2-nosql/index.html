Demands of Modern Database Applications

  • Massive data amount (Big Data)
  • Rapidly changing data types
  • Modern development approaches: agile software development, etc.
  • New use-cases: geospatial applications, ubiquitous computing, graph-based applications, data versioning, etc.
  • Non-traditional database requirements: consistency, reliability, scalability, latency, etc.

NoSQL Database Types

  • Key-value Stores: Redis
  • Wide-Column Stores (spaltenorientierte Datenbank): Cassandra, HBase
  • Graph Databases: Neo4j
  • Document Databases: CouchDB, MongoDB, ArangoDB

Key-value Stores

Redis

Key-value Store

  • Redis = remote dictionary server
  • Most popular key-value store
  • Used by Twitter, GitHub, Snapchat, StackOverflow
  • BSD License, written in ANSI C
  • Supports clusters, geospatial queries
  • Configurable for ACID

Example

Usually (extended) JSON documents

$ redis-cli
redis 127.0.0.1:6379> set mykey somevalue
OK
redis 127.0.0.1:6379> get mykey
"somevalue"

Wide-Column Stores

Wide-Column Stores

  • German term Spaltenorienterte Datenbank used for different approaches
  • Names and format of columns can vary from row to row
  • Multi-dimensional key-value store
  • Google BigTable in 2004
  • Cassandra used by Facebook (Inbox Search), Netflix, Wikipedia (Content API)

Graph Databases

Graph Databases

  • Data representation using nodes (entities) and edges (relationships)
  • Nodes with additional information (properties)
  • Neo4j used by Airbnb (Data Exploration), NASA (Knowledge Representation), Deutsche Bahn (Configuration Management)

Recap: ACID

  • Atomicity: all or nothing
  • Consistency: always a valid state
  • Isolation: concurrency control
  • Durability (Persistenz): survice crashes

CAP-Theorem

(Brewers Theorem)

In a distributed data storage it is impossible to simultaneously provide more than two out of the following three guarantees: consistency, availability, partition tolerance.

CAP

  • Consistency: every read receives the most recent write or an error
  • Availability: every request gets a non-error response
  • Partition Tolerance: system reliable even with dropped messages

BASE

  • Basically Available
  • Soft State: not always consistent
  • Eventual Consistency: becomes consistent at some time
